# Build HTML Slides

Create an offline-portable bundle containing one `.html` presentation, its local assets folder when needed, and a sibling `-notes.md` presenter guide. Treat each viewport as a designed slide, not a scrolling webpage.

Every deck must produce the sibling `OUTPUT-notes.md` presenter guide, including Quick Draft work. Speaker notes are part of the deliverable, not an optional polish step.

Validation depth and design quality are separate. Quick Draft may reduce research and review breadth, but it must still look intentionally designed, use a topic-specific theme, preserve meaningful imagery, and pass the minimum rubric in `references/quality-bar.md`. Retain all required render profiles, but use adaptive AI review: inspect `normal` for every slide and add stress profiles only for the cover, closing, slides marked `data-visual-critical="true"`, requested responsive targets, or automated warnings. Review three or four slides per vision call while keeping one observation and verdict per slide. A contact sheet, DOM metrics, or source-only review never counts as per-slide coverage. Never begin AI review while the automated text/control/image geometry gate is failing. When text is in scope, trace every text region against its intended box, cell, button, badge, column, or safe area and pass `text_bounds`. Never ship the starter template's demo palette, copy, or three-card composition as the deck's visual identity.

## Validation Mode Gate

Choose the validation mode before substantial research, asset generation, implementation, or rendering.

- For an existing-deck revision, default to **Edit Only**. When the user merely asks to change, fix, add, remove, restyle, or adjust something, make the requested edit and do not run new render captures, validators, visual QA, browser tests, rubric scoring, or multi-agent review. Do not ask the Quick Draft versus Full Validation question for an ordinary revision.
- Upgrade an existing-deck revision only when the user explicitly asks to validate, test, inspect, review, check the result, make it final/production-ready, or uses precision language such as 꼼꼼히, 정밀하게, 철저히, 검토까지, 검증까지, final, polished, thorough, verify, validate, test, or equivalent. Use Full Validation for precision/final requests; use only the specifically requested check when the scope is narrower.
- Edit Only may update sibling presenter notes when the requested change affects slide titles, claims, order, or talk track, but it must not create a new visual-review pass. Any existing render manifest becomes stale after an HTML edit and must not be presented as current evidence. State plainly at delivery that the revision was not re-rendered or revalidated.
- Use **Quick Draft** without asking when the user says 초안, 라이트, 빠르게, 샘플, 러프, 간단히, quick, draft, light, sample, or equivalent. Optimize for turnaround. Keep the deck functional and readable, but avoid exhaustive polish.
- Use **Full Validation** without asking when the user says 최종, 꼼꼼히, 정밀 검증, 발표용, 배포용, 완성본, final, polished, production-ready, presentation-ready, thorough, or equivalent.
- If the user does not indicate speed or validation depth, ask exactly once before starting: “빠른 초안은 핵심 구성과 기본 동작만 확인해 빨리 만들 수 있습니다. 정밀 검증은 전 슬라이드 렌더링·출처 확인·다중 검토 때문에 시간이 오래 걸립니다. 빠른 초안과 정밀 검증 중 어느 쪽으로 진행할까요?” Do not begin expensive research, ImageGen, full rendering, or multi-agent review until the user chooses.
- Do not ask again after a mode has been chosen in the current request. An explicit later instruction overrides the earlier mode.

When Full Validation is selected, run `node scripts/render_slides.js --check` immediately, before substantial research or implementation. This preflight must load Playwright, launch Chromium, render a page, and capture an image successfully. If it fails, stop and report the exact missing component or launch error. Ask the user whether to install the required validation tooling; never install Node.js, Playwright, Chromium, or system browser dependencies without that explicit approval. After approval, use the environment's existing package manager and Playwright installer, then rerun `--check`. Do not proceed under or claim Full Validation until the preflight passes. A later tool failure invalidates the preflight and requires the same disclose, ask, install, and recheck sequence.

### Quick Draft

- Build the requested story and core interactions, but keep research and asset sourcing intentionally limited. Reduce validation breadth, not visual ambition: read `references/audience-story-routing.md`, route the story for the named or inferred audience, choose a topic-specific theme, vary composition families, and use supplied/local assets first. Verify only claims that are central, unstable, or high-risk.
- Read `references/quality-bar.md` before design, but calculate its 24-point score only once after the render/fix loop is settled. Require at least 16/24 with no zero before delivery.
- Create `OUTPUT-notes.md` from `assets/speaker-notes-template.md` with one numbered section per slide. Keep notes concise, but include purpose, natural talk track, emphasis, transition, and source/caveat fields.
- Run `scripts/validate_deck.py OUTPUT.html`, `scripts/validate_speaker_notes.py OUTPUT.html OUTPUT-notes.md`, and `scripts/validate_interactions.py OUTPUT.html`. Run `scripts/source_cache.py OUTPUT.html sources.json --update`, complete only `needs-review` entries, then run `--check`; unchanged image hashes retain prior source research. Run `scripts/validate_source_locality.py` as a Korean-market link heuristic when the deck contains audience-specific prices, dates, availability, schedules, regulation, or market data; manually verify the target region for every locale. Fix errors; do not expand into an exhaustive audit.
- Run `node scripts/render_slides.js OUTPUT.html REVIEW_DIR --mode quick`. It creates the default `normal`, `short`, and `zoom150` evidence while disabling motion only in Playwright; final HTML animations remain intact. If the automated gate fails, fix its text, control, or image geometry findings and rerun before opening captures. Read `references/slide-by-slide-review.md`; use generated `review_batches` to inspect three or four slides per vision call and inspect only each slide's `required_ai_profiles`, while writing one observation/verdict per slide. Run `scripts/validate_visual_review.py OUTPUT.html REVIEW_DIR/review.json`. After a fix, use `--slides N --change-type text|image|navigation|all`; the renderer refreshes the changed slide and neighbors while reusing fingerprint-matched evidence. When settled, run `--finalize`, calculate the quality score once, and validate again.
- Skip tablet/mobile profiles, exhaustive link checking, full contrast sampling, fullscreen/print testing, and Multi-Agent Final Review unless the user explicitly requests them. Add `--responsive` only for requested mobile/tablet support.
- Clearly label delivery as a quick draft and list the checks actually performed. Never imply full validation.

### Full Validation

Follow the complete workflow, visual QA, source review, and Multi-Agent Final Review below.

## Full Workflow

1. Inspect supplied references for stage, visual grammar, slide taxonomy, navigation, density, and defects. When the user supplies successful prior decks, treat them as a quality benchmark: extract reusable traits such as visual rhythm, image density, claim style, composition range, and closing payoff without copying their content or identity. Preserve the requested character without copying broken encoding or brittle layout.
2. Establish the story before coding. Read `references/quality-bar.md` and `references/audience-story-routing.md`. Identify the named or inferred audiences, decision owner, implementation owner, desired room outcome, baseline knowledge, detail tolerance, and likely objections. Create an audience model and a slide-role table with one main claim, `primary_audience`, `secondary_audience`, `audience_question`, `decision_job`, `detail_level`, `why_now`, composition family, dominant visual, and transition per slide. Order the deck by the room's shared decision path rather than a generic topic outline: earn attention with common stakes and decision context, surface the recommendation or ask before attention fragments, and defer specialist depth until it answers an established question. Mixed executive/business/developer audiences often benefit from an early problem or `AS-IS / TO-BE` contrast followed by impact and decision, with architecture and implementation later, but treat this only as an example and choose the sequence from the actual audience and purpose. Identify market, jurisdiction, and locale before research. For prices, launch dates, availability, retail, regulation, events, schedules, and market metrics, read `references/source-locality.md` and use target-region official/local sources first. For Korean decks, read `references/korean-copy.md`; write as a Korean presenter would speak, not as translated marketing copy. Prefer established Korean names and ordinary Korean words, keeping English only for official names, standards, acronyms, or a small secondary label.
3. Choose theme and typography deliberately. Follow explicit direction; otherwise infer a fitting direction from subject, audience, and tone. Read `references/theme-playbook.md`, `references/theme-gallery.md`, and `references/style-presets.md`. Write a one-line theme contract and choose at least five composition families for a typical 8-15 slide deck. Test the heaviest title and densest body slide in the actual writing system before committing.
4. Decide whether systems, integrations, deployment, security boundaries, or flows need an architecture diagram. When they do, read `references/architecture-diagrams.md` and include the diagram even if the user did not explicitly ask.
5. Make an asset plan: official logo/wordmark, official product/device photography, factual product or people imagery, supplied references, and original atmosphere art worth generating. For a named product or brand, place the official logo prominently on the cover when one exists and use official device/product imagery on at least one explanatory slide when available. For games and animation, prefer official sites, publisher/studio media pages, press kits, wallpapers, and key art before fan work or generic imagery. For private/internal decks, suitable fan art may be used when it materially improves the story even when formal permission evidence is unavailable; keep the creator/source URL when discoverable, preserve signatures and watermarks, credit the creator, and never imply official endorsement. Public, commercial, or broadly distributed work still requires a rights review or a safer replacement. Use ImageGen for as many distinct scenes as the story justifies; do not cap generated imagery at one asset. Convert every raster asset to WebP before it enters the HTML bundle; keep true vector logos, icons, and diagrams as SVG. Maintain sibling `sources.json` with each local raster path, SHA-256, source kind, source URL when applicable, verification time, credit, and status. Run `source_cache.py --update` before research on a revision and revisit only entries marked `needs-review`.
6. Copy `assets/presentation-template.html` to the output path for its runtime and structural shell. Replace every demo slide, demo token, and default composition; the template is not an acceptable visual direction. Preserve the `visualViewport`-aware navigation/runtime unless another interaction model is requested.
7. Use UTF-8 and semantic `<section class="slide" data-title="…">` elements. Give the initial slide `active`; let the runtime reconcile the URL hash.
8. Build editable diagrams, icons, charts, and decoration with HTML/CSS or inline SVG. Escape user-provided text.
9. Create `OUTPUT-notes.md` from `assets/speaker-notes-template.md`. Include one numbered section per slide with the exact slide title and at least five bullets covering purpose, natural 30–90 second talk track, emphasis cue, transition, and source/caveat reminder. Translate the field labels into the deck language when appropriate. Add delivery guidance rather than repeating slide copy. This is mandatory in both Quick Draft and Full Validation modes.
10. In Full Validation mode, run `scripts/validate_deck.py OUTPUT.html`, `scripts/validate_speaker_notes.py OUTPUT.html OUTPUT-notes.md`, `scripts/validate_image_reuse.py OUTPUT.html`, `scripts/validate_interactions.py OUTPUT.html`, `scripts/source_cache.py OUTPUT.html sources.json --update`, and `scripts/validate_source_locality.py OUTPUT.html`. Complete only new/changed `needs-review` source records, then require `scripts/source_cache.py OUTPUT.html sources.json --check` to pass. Treat source-locality output as a Korean-market link heuristic, not factual proof; manually verify prices, dates, availability, schedules, regulation, and market claims against the selected target region for every locale. Fix every error and inspect all warnings.
11. In Full Validation mode, run `node scripts/render_slides.js OUTPUT.html REVIEW_DIR --mode full`. Default evidence remains exactly `normal`, `short`, and `zoom150`; add `--responsive` only when mobile/tablet support was requested. Validation-only CSS disables animation during capture and waits two frames, while the final HTML retains its authored motion. The renderer must block AI review when scoped text bounds, control geometry, or image load/aspect/resolution checks fail. Fix those failures before opening captures. Use generated batches of at most four slides per vision call; inspect `normal` for ordinary slides and the generated `required_ai_profiles` for covers, closings, explicit `data-visual-critical="true"` slides, requested responsive targets, and warning-triggered slides. Keep one observation, one scoped check set, and one status per slide. After a fix, rerender only the changed slide and immediate neighbors with `--slides N --change-type text|image|navigation|all`. Global style/runtime changes automatically force a full rerender. Run `scripts/validate_visual_review.py OUTPUT.html REVIEW_DIR/review.json`; it validates PNGs, hashes, geometry, adaptive routing, batch membership, incremental lineage, and current HTML/local-asset fingerprints without a second full screenshot pass. Use contact sheets only for deck-wide rhythm. Read and complete `references/visual-qa.md`, then test arrows, Page Up/Down, Space, Home, End, hash navigation, buttons, direct page input, resize, fullscreen, print, and boundaries.
12. Run the multi-agent final review below. Apply actionable feedback with scoped incremental renders. Once the deck is settled, run `node scripts/render_slides.js OUTPUT.html REVIEW_DIR --finalize`, calculate `quality_score` once, add required final cross-reviews, and run the prescribed validators one final time. `--finalize` must fail when reviewed source fingerprints no longer match.

## Theme Direction

- Translate theme direction into typography, surfaces, shapes, imagery, density, and motion—not only color.
- When no direction is given, infer two or three adjectives internally and implement one cohesive system without asking.
- Vary paper, dark, tinted, and image-led surfaces when useful while retaining the same grid, type roles, accents, and navigation.
- Never copy a recognizable deck or artist pixel-for-pixel. Extract high-level attributes.

## Visual Assets

Choose the lightest effective method.

1. Use CSS/inline SVG for editable abstract geometry, diagrams, charts, and generic icons. Never redraw a real logo when an official asset is available.
2. Inspect supplied images first. Copy project assets beside the deck in a descriptive assets folder; never point to a temporary path.
3. For every named brand, product, game, event, institution, or project, search for its official logo or wordmark first. Acquire the official SVG or transparent PNG when available; use SVG directly for true vector artwork or convert a raster PNG source to WebP for the deliverable. Preserve aspect ratio and clear space, and never approximate it with typed text or a homemade icon. If unavailable, use a neutral label and record the limitation in notes.
4. Treat cover branding as part of the message, not a footnote. For a brand/product deck, the first slide must show the official logo or wordmark at a readable size unless the user explicitly asks for a logo-free cover. A typed brand name does not count when an official mark exists.
5. For factual photos, products, posters, characters, maps, screenshots, and places, search the web. Prefer official press kits/sites, institutional sources, Wikimedia Commons, or licensed stock. Download permitted assets locally, record source URLs in HTML comments, add meaningful alt text, and visibly credit when appropriate. Do not hotlink unless requested. A product-spec or hardware slide should use an official device image when available instead of a homemade approximation.
6. For a game or animation title, search its official site and the publisher/studio's media page, press kit, wallpaper downloads, key art, title art, character art, and screenshots before using generic stock or generated substitutes. Official availability does not automatically grant every reuse right: check the page terms or media guidance, keep the source URL, preserve logos/faces/composition edges, avoid spoilers, and convert the selected raster locally to WebP.
7. Prefer official game/anime art, wallpapers, press assets, and creator-hosted originals. For a private or team-only deck, fan art may be used without making formal permission or license evidence a hard validation requirement. Keep the artist name and source URL when discoverable, preserve signatures/watermarks, credit the artist, and avoid repost aggregators or falsely presenting the work as official. For public, commercial, client-facing, or broadly distributed decks, verify the intended reuse rights or replace the asset with official, licensed, supplied, or original work.
8. If the user requests ImageGen or original raster artwork genuinely improves the deck, use `$imagegen`. Generate multiple distinct assets when separate slides need different scenes, moods, or visual jobs. Strong candidates are atmospheric chapter art, conceptual metaphors, editorial illustrations, lifestyle scenes, and bespoke backgrounds with text-safe space. Do not generate substitutes for specific real products, screenshots, logos, or factual events when official sourcing is appropriate.
9. Promotional, product, travel, entertainment, lifestyle, and event decks should normally contain meaningful raster imagery across most of the story, not only the cover. Mix official/sourced photography, supplied imagery, and justified generated art without repeating one hero image everywhere.
10. For image edits, use `$imagegen` edit mode. Inspect with `view_image`, preserve requested invariants, save non-destructively, and copy the selected output into the project.
11. Design crops for the slide. Use intentional focal points and text-safe space; avoid embedded text where HTML text is more accurate and accessible.
12. Optimize assets near displayed size. Every raster image in the deliverable must be WebP, including sourced photos, screenshots, generated art, transparent cutouts, thumbnails, and video poster frames. Convert PNG, JPEG, GIF, TIFF, BMP, and AVIF inputs before referencing them. Keep SVG only for genuinely vector logos, icons, and editable diagrams. Use `object-fit` and `object-position`; never stretch.
13. Do not place a non-16:9 key visual, official title art, transparent logo, product shot, game/anime art, poster, or other edge-important image into a 16:9 frame with `object-fit: cover`. Use a contained foreground visual, a decorative blurred/covered backdrop plus contained foreground, or a redesigned composition. A cropped key visual is a failed draft unless every cropped edge is explicitly decorative.
14. Avoid image-reuse abuse, not all reuse. A factual title/product image may appear on at most two slides when it creates useful continuity—for example, once as a cover or game-detail hero and once as a visibly smaller lineup thumbnail. Mark both occurrences with the same `data-repeat-ok="continuity-id"` and set `data-image-role="hero|detail|thumbnail"`; one occurrence must be `thumbnail`, the other `hero` or `detail`. Never approve repeated full-bleed backgrounds, hero+hero, background+anything, more than two slides, or repetition used only to fill space. Reviewers must confirm that the smaller use is subordinate and that each appearance serves a different narrative job.
15. Put raster slide imagery in `<img>`, `<picture>`, or SVG `<image>` elements with local paths. Prefer `.slide-media` for background/layered media and `.slide-content` only for meaningful foreground media such as logos, diagrams, screenshots, or contained key visuals. Do not hide raster assets in global CSS `background:url(...)`. CSS may use gradients, shapes, and inline SVG theme decoration. This keeps asset ownership, layering, alt text, and reuse validation deterministic.

## Architecture Diagrams

- Choose context, component, request/data flow, sequence, deployment/topology, trust boundary, or before/after based on the question.
- Prefer editable SVG or structured HTML/CSS for topology, boundaries, arrows, protocols, and labels. ImageGen may supply an illustrative isometric environment, metaphor, or textured backdrop when it improves comprehension, but overlay factual components and connections in editable HTML/SVG. For conceptual rather than factual architecture, a generated diagram is acceptable if it is clearly labeled as illustrative.
- Confirm components and relationships from supplied or authoritative material. Mark uncertainty as `TBD`, `assumed`, or `not specified`.
- Keep one architectural message per slide. Use progressive detail instead of one everything-diagram.
- Label arrows with verbs/protocols and distinguish sync, async, replication, control plane, and data plane consistently.
- Show system, network, region, cluster, namespace, account, or trust boundaries as labeled containers.

## Design Contract

- Use the viewport-safe shell in `references/style-presets.md`. When a logical 1280×720 or 1920×1080 stage is useful, scale it uniformly from `window.visualViewport.width/height` with `document.documentElement.clientWidth/clientHeight` only as fallback. Do not rely on `window.innerWidth` / `window.innerHeight` alone; browser zoom, display scaling, side panels, and mobile viewport chrome must not cut the stage.
- Maintain hierarchy: compact kicker, 36–60px title at 1280×720, 18–24px body, restrained support text.
- Choose typography intentionally. Prefer a display/body pair or one variable family with distinct weights. Verify Korean glyph quality, punctuation, numerals, and Latin/Korean mixing. Do not rely on a Latin display face whose Korean fallback looks unrelated. Package local webfonts when licensing permits; otherwise use a coherent system stack and record it in notes.
- Tune spacing by role. Tighten display titles only enough for cohesion; keep Korean body text comfortable; avoid excessive mono tracking in long kickers, awkward single-character wraps, widows, and stranded punctuation.
- Prefer columns, cards, comparisons, timelines, diagrams, image crops, and short tables over dense prose.
- Center numerals, icons, badges, CTA contents, and control labels with grid/flex, not padding or line-height guesses. Separate card buttons from compact CTA buttons so card rules such as `align-items:flex-start` or `justify-content:space-between` cannot leak into CTAs. Wrap a CTA's label and icon in one centered inline-flex group; center the combined bounding box, not the label alone. Verify horizontal and vertical center from rendered bounding rectangles, then apply at most a 1–2px optical correction. Test text-only, text+arrow, `1`, `2`, `8`, `10`, and the maximum page count. Use `font-variant-numeric: tabular-nums`. Keep page input, slash, and total baseline and vertical center aligned at every digit count.
- Use whitespace for hierarchy, not unfinished areas. Except deliberate hero/chapter slides, make the main composition occupy roughly 65–85% of the safe canvas. If a lower or side third is empty, enlarge or redistribute useful content, add a meaningful visual/data element, or change layout. Do not add meaningless filler.
- Use coherent CSS tokens, readable contrast, and visible focus states. Treat contrast as a rendered-geometry requirement, not a palette intention. Body text and controls need at least 4.5:1 contrast; large display text needs at least 3:1. Sample the actual pixels behind text on images, gradients, diagonal shapes, and overlays. Text or icons must not straddle light/dark or same-color boundaries unless an opaque backing surface, outline, or scrim keeps the full glyph readable. Same-color or near-same-color text/background is a blocking defect.
- Require exactly one direct `.slide-media` child and one direct `.slide-content` child on every slide. Use `.slide-media { position:absolute; inset:0; z-index:0; pointer-events:none }`, `.slide-content { position:relative; z-index:2 }`, and `.slide { isolation:isolate }`; keep controls above the stage. A foreground image, SVG, pseudo-element, diagram, or decorative shape must never cross in front of or obscure a title, body, label, source, or control. If art needs to overlap the composition, crop or mask it behind a text-safe region; do not solve the collision by reducing text opacity.
- Animate with opacity and a small transform around 300–600ms. Respect `prefers-reduced-motion`.
- Keep controls discoverable without dominating the slide. Preserve the template's fixed order: previous icon, editable current-page number, slash, total, next icon, fullscreen icon. Default to a compact bottom-right cluster, roughly 180–260px wide and 44–56px high. Theme it through `--nav-surface`, `--nav-border`, `--nav-ink`, `--nav-muted`, `--nav-accent`, and `--nav-hover`; do not change the interaction structure for visual novelty. Keep icon buttons quiet until hover/focus and keep the numeric input visually integrated into the counter rather than boxed as a separate card. Keep the progress indicator separate. Do not display the current slide title, subtitle, section label, or truncated text inside the control cluster; the active slide already provides context, while accessible names can carry hidden labels. Add visible nav text only when the user explicitly requests it and mark the exception with `data-nav-title-ok`. Move keyboard hints away from the controls. Use a larger centered bar only when the user requests it or accessibility requirements demand it.
- Build the page counter as three stable cells: current-page input, separator, and total. Give them the same explicit height, `display:grid`/flex centering where supported, `line-height:1`, zero vertical padding, and tabular numerals. Keep the current and total numeral centers on the counter's vertical center within 1.5px; do not align them with ad hoc top padding or transforms.
- Make every element that looks interactive actually work. Use `<button>` for in-deck actions and `<a>` for external destinations; never style a passive `<div>` as a CTA. Cards, chips, arrows, logos, and closing CTAs that imply navigation must respond to click/tap, show a focus state, and expose an accessible label. In-deck controls use `data-goto`, `data-next`, or `data-prev`; external links use a real `href` and open safely when appropriate.
- Do not invent redundant utility CTAs such as “back to cover,” “last slide,” “restart,” or “view the final slide” when persistent previous/next and direct-page controls already exist. Reserve in-slide buttons for story-relevant jumps, audience choices, demos, or real external actions. Add a utility jump only when the user explicitly requests it or persistent navigation is unavailable; mark an intentional exception with `data-utility-nav-ok`.
- Direct navigation must use numeric `<input id="pageInput">`: select on focus, navigate on Enter/change, clamp to `1..N`, restore after invalid input, and suppress slide shortcuts while editing.
- Update hashes (`#1`, `#2`, …) and add accessible labels. Ignore navigation keys in inputs, textareas, selects, and editable regions.
- Include print CSS when export or printing is plausible.

## Content Fit and Integrity

Fit by editing, not shrinking everything. Shorten copy, split a slide, simplify a diagram, crop imagery, or change layout before reducing type. Avoid body below 16px and code below 13px.

Keep UTF-8 end-to-end. Treat replacement characters and mojibake as blocking defects. Validate current claims with authoritative sources and keep citations in visible credits, notes, or nearby HTML comments.

Before delivery, compare rendered geometry rather than trusting CSS intent. Check true centers and baselines, repeated padding, official logo aspect ratio and clear space, title/body tracking, crop focal points, occupied-area balance, and top-heavy or unfinished slides. Follow `references/visual-qa.md`.

Score the settled renders once, after `--finalize`, with `references/quality-bar.md`. Quick Draft requires at least 16/24 with no zero. Full Validation requires at least 20/24 with no dimension below 2. Do not recalculate the rubric during each fix iteration. Structural validity, source accuracy, or responsive fit cannot compensate for generic art direction, repeated layouts, sparse imagery, or a weak closing.

Run the Resolution Safety Gate from `references/style-presets.md` and the per-slide audit from `references/slide-by-slide-review.md` before showing any draft or final deck. Verify no meaningful text, logo, title art, key visual, diagram, screenshot, badge, or slide control crosses the viewport. Verify key visuals are not cropped unless marked decorative. Default validation retains normal desktop, short browser height, and 150% zoom evidence. AI always inspects normal; stress profiles are routed adaptively from explicit critical markers and automated warnings. Mobile/tablet evidence and AI review are opt-in. No slide may inherit a pass from another slide or from a contact sheet.

## Multi-Agent Final Review — Full Validation Only

Skip this section in Quick Draft mode. In Full Validation mode, use at least two independent reviewer agents. Use three for high-stakes, long, technical, or image-heavy decks. When the user explicitly requires multi-agent review, unavailable reviewers are a blocker rather than a disclosure-only fallback.

1. Assign visual reviewers contiguous ranges of at most six slides, but execute the renderer's `review_batches` as vision calls of at most four slides. Give each reviewer only the full-size captures listed in each slide's `required_ai_profiles` plus relevant source assets. Record both a readable `reviewer` label and stable `reviewer_ref`. Require one concrete observation and one scoped verdict per slide, not per capture. Source-only, geometry-only, and contact-sheet-only review are invalid.
2. After `--finalize`, assign one presentation editor to calculate the quality bar once. Record dimensions, total, three weakest slides (or all slides when fewer than three), and concrete findings on opening promise, audience-specific information order, decision timing, story flow, claim strength, audience fit, evidence, source locality, omissions, transitions, closing payoff, and notes usefulness.
3. For technical decks, assign a third reviewer to architecture correctness; for promotional decks, assign a third reviewer to brand/asset provenance and persuasion ethics.
4. Give each reviewer raw deliverables and the original request, not the creator's diagnosis. The renderer disables motion only in its Playwright context and captures after two animation frames; never remove animation from the delivered HTML. Ask for slide number, severity, visible evidence, and a concrete fix.
5. Consolidate duplicates, reject unsupported suggestions, and apply every valid high/medium-severity fix. Do not spend the review on tiny spacing comments while generic theme, repeated composition, weak imagery, or story gaps remain. The creating agent owns the edits; reviewers provide feedback and may re-review the revised artifacts.
6. In final phase, independently cross-review the cover, closing, and marked critical slides once each. The cross-reviewer `reviewer_ref` must differ from the primary reviewer and record current capture hashes. During fixes, run only validators relevant to the changed text, image, or navigation, and rerender the changed slide plus neighbors. After `--finalize`, run the complete structural and deliverable validators once. Unresolved high/medium findings or a failing final score block delivery.
7. Keep a compact review disposition table in the work log: reviewer, slide, severity, finding, decision, edit made, and re-review status.
